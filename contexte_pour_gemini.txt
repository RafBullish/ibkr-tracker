  + hooks uniquement.

### Flux de données (global)
Source de vérité unique = store Zustand (src/store/useStore.js), persisté dans
localStorage (clés ibkr_u_o/c/f/j/s, écritures debouncées et reference-gated).
5 slices : openPositions, closedTrades, cashFlows, journalEntries, settings.

- Entrée des données :
  (a) import IBKR Flex via services/flexApi.js + api/flex/sync.js
      -> actions SYNC_FLEX / SYNC_IBKR ;
  (b) import CSV/JSON via utils/ibkrParser.js + utils/ibkr/*
      -> action IMPORT_DATA ;
  (c) saisie manuelle via AddTradeModal
      -> ADD_POSITION / ADD_CLOSED_TRADE.
- Transformations : dispatch(action) -> reducer.js (pur, (state, action) =>
  state) ; versionnage de schéma via migrations.js.
- Sortie vers l'UI : sélecteurs granulaires (useOpenPositions, useClosedTrades,
  useSettings…) ; les ~31 hooks custom dérivent les métriques
  (usePortfolioMetrics, useTradingMetrics, useEquityHistory…).
- Données live externes : hooks (useFx, useMarketQuotes, useCalendarFeeds,
  useApiStatus) -> routes /api/* -> APIs tierces (Finnhub, Yahoo, CBOE,
  Twelve Data).

    "gates gates gates gates gates gates gates gates gates gates gates gates"

/* Mobile <768px : grid-template-columns: 1fr ; rows: repeat(12, 200px) ; modules empilés */

POINT D'ATTENTION 4K / zoom 90 % (Phase C.3.0) : les colonnes sont en fr
(scalables), mais les 5 hauteurs de rows sont codées en pixels fixes
(240/160/130/200/200). Sur un écran 4K, la grille s'étire en largeur mais les
rangées restent à hauteur fixe — c'est le point dur du futur scale. Aucune
unité auto, aucun minmax(), aucun clamp(). Toute refonte responsive 4K devra
remplacer ces px par fr / minmax() / clamp().

## 5. Features implémentées
================================================================

Import / synchronisation broker
- Import IBKR Flex Query (services/flexApi.js + api/flex/sync.js) avec
  dédoublonnage par signature de trade/flux.
- Parsing CSV IBKR multi-sections (utils/ibkr/{csvReader,sections,closedTrades,
  merge}.js, ibkrParser.js).
- Import/export JSON de sauvegarde (IMPORT_DATA).
- Saisie manuelle de trades (AddTradeModal), agrégation multi-lots avec prix
  moyen pondéré.
- Système de migration de schéma versionné (migrations.js).

Positions & historique
- Positions ouvertes : prix live, clôture partielle/totale (CLOSE_POSITION).
- Historique des trades clos, détection automatique du motif de sortie
  (utils/trades/detectExitReason.js), confirmation/override manuel.
- Distribution d'historique (HistoryDistribution.jsx), heatmap calendaire PnL.

Analyse & calculs
- Métriques portefeuille & trading (usePortfolioMetrics, useTradingMetrics,
  win-rate, etc.).
- Greeks (utils/greeks.js, greeksApi.js) et pricing Black-Scholes
  (utils/options/blackScholes.js).
- Risque (utils/risk.js, RiskMatrix), kill-switch journalier, capital
  disponible.
- Attribution de performance, analyse de chaîne d'options (chainAnalytics.js),
  IV rank, skew, internals de marché, sectors.
- Conversion FX USD/CHF (manuelle ou auto, bannière de taux périmé).

Marché & calendriers
- Cotations actions (Finnhub/Yahoo), chaîne d'options (CBOE).
- Calendrier de résultats et calendrier macro 2026, TickerTape live,
  Pre-Market Briefing.
- Gates « Sniper » paramétrables (SniperMetaEditor, useSniperGates,
  useNextGate).

Journal & insights
- Journal de trading (CRUD), pages Analytics et Calendar.

UI / Shell
- Shell « Institutional Terminal » top-down (CommandBar, TickerTape, StatusBar,
  BottomNav mobile).
- Command Palette (Cmd+K), cheatsheet clavier (Cmd+/), navigation Cmd+1-9.
- 2 thèmes (midnight/daylight), mode colorblind, reduced-motion, ThemeSwitcher.
- Tables virtualisées, skeletons de chargement, toasts, ErrorBoundary,
  tooltips/modales Radix.

const NAV_PATHS = [
  '/dashboard', '/trading/positions', '/trading/history',
  FEATURE_GREEK_CENTER ? '/trading/greeks' : null,
  '/trading/chain', '/insights/analytics', '/insights/calendar',

function SubNav({ pathname, navigate }) {
  let tabs = [];
  if (pathname.startsWith('/trading')) {
    tabs = [
      { path: '/trading/positions', label: 'Positions' },
      { path: '/trading/history', label: 'Historique' },
      ...(FEATURE_GREEK_CENTER ? [{ path: '/trading/greeks', label: 'Greeks' }] : []),
      { path: '/trading/chain', label: 'Chain' },
    ];
  } else if (pathname.startsWith('/insights')) {
    tabs = [
      { path: '/insights/analytics', label: 'Analytics' },
      { path: '/insights/journal', label: 'Journal' },
      { path: '/insights/calendar', label: 'Calendrier' },
    ];
  } else if (pathname.startsWith('/settings')) {
    tabs = [
      { path: '/settings/general', label: 'General' },
      { path: '/settings/import', label: 'Import' },
      { path: '/settings/api', label: 'API' },
    ];
  }
  if (tabs.length === 0) return null;
  return (
    <nav className="sub-nav" aria-label="Sous-navigation">
      {tabs.map((tab) => (
        <button key={tab.path} type="button" className="sub-nav__tab"
          data-active={pathname === tab.path || undefined}
          onClick={() => navigate(tab.path)}>
          {tab.label}
        </button>
      ))}
    </nav>
  );
}

export default function AppShell() {
  const [cmdOpen, setCmdOpen] = useState(false);
  const [cheatOpen, setCheatOpen] = useState(false);
  const { pathname } = useLocation();
  const navigate = useNavigate();
  const isMobile = useMediaQuery('(max-width: 767px)');

  useEffect(() => {
    const handler = (e) => {
      if ((e.metaKey || e.ctrlKey) && e.key === 'k') { e.preventDefault(); setCmdOpen((p) => !p); return; }
      if ((e.metaKey || e.ctrlKey) && e.key === '/') { e.preventDefault(); setCheatOpen((p) => !p); return; }
      if ((e.metaKey || e.ctrlKey) && e.key >= '1' && e.key <= '9') {
        e.preventDefault();
        const target = NAV_PATHS[parseInt(e.key, 10) - 1];
        if (target) navigate(target);
      }
    };
    document.addEventListener('keydown', handler);
    return () => document.removeEventListener('keydown', handler);
  }, [navigate]);

  useEffect(() => {

## 8. Évaluation honnête (code review senior)
================================================================

### 3 points forts
1. Couche state exemplaire. Zustand + reducer pur + sélecteurs granulaires +
   système de migration versionné + persistance localStorage debouncée et
   reference-gated. C'est propre, testable et au-dessus du niveau attendu.
2. Logique métier bien extraite de l'UI. ~35 modules utils/ + 31 hooks isolent
   les calculs (Black-Scholes, Greeks, parsers IBKR, attribution, FX) hors des
   composants. La séparation logique/présentation est réelle dans la couche JSX.
3. Hygiène de production sérieuse. CSP/HSTS/anti-clickjacking, allowlist CORS
   sans fallback permissif, rate limiting, secrets exclusivement côté serveur,
   feature flags, ErrorBoundary, code-splitting (lazy + manualChunks),
   accessibilité (skip-link, reduced-motion, colorblind). Documentation in-code
   remarquable.

### 3 points faibles / dette technique
1. Zéro test sur du code qui calcule de l'argent. calculations.js (543 l.),
   greeks.js, blackScholes.js, parsers IBKR : aucune vérification automatique.
   C'est le risque n°1.
2. CSS hors de contrôle. 12 feuilles, ~16k lignes, couches versionnées
   superposées (v3-components 5494 l., dashboard.css 1286 l. probablement morte,
   ~150 alias "legacy" dans tokens.css), cascade dépendante de l'ordre d'import.
   Deux échelles typographiques rivales. C'est le frein direct à toute refonte
   visuelle.
3. Dashboard à moitié construit + pages géantes. 7/12 modules sont des stubs en
   empty state ; Calendar.jsx (1234 l.), Positions.jsx (880 l.), History.jsx
   (787 l.) mélangent logique et UI. Aucun linter/formatter/CI, pas de
   TypeScript malgré des structures de données aux clés cryptiques.

### 3 améliorations prioritaires (avant la Phase C.3.0)
1. Consolider le CSS d'abord. Supprimer les feuilles mortes (dashboard.css),
   auditer v3-components.css, fusionner vers tokens + 2-3 couches, unifier la
   typo sur une seule échelle. Une refonte visuelle par-dessus 12 cascades
   fragiles échouera.
2. Trancher le sort des 7 modules stub. Soit câbler les données, soit les
   retirer de la grille. Auditer l'UX d'empty states permanents est du temps
   perdu.
3. Tester la "money math" (calculations, greeks, blackScholes, parsers IBKR) et
   ajouter ESLint + Prettier ; passer la grille dash-grid en fr/minmax()/clamp()
   pour le 4K.

### Niveau d'autonomie du développeur
Évaluation : INTERMÉDIAIRE (avec exécution fortement amplifiée par l'IA).

Justification : la base de code montre une maturité architecturale supérieure à
un débutant — store/reducer/migrations propres, plugin Vite maison, sécurité
prod soignée, design-system tokenisé. Le développeur maîtrise clairement le
domaine métier (options, Greeks, IBKR Flex). Mais plusieurs signaux pointent une
exécution massivement assistée par IA plutôt qu'une autonomie "confirmée" :
volume (~40k lignes) et niveau de finition incohérents avec ~3-5 mois
d'apprentissage en solo, style de commentaires extrêmement dense et méthodologie
"brick / Sprint" très structurée, et surtout les angles morts classiques d'un
dev encore en montée en compétence (aucun test, aucun linter, pas de TS, CSS
legacy non nettoyé, 7 modules laissés en stub). En clair : capable de DIRIGER
une architecture solide et de juger le résultat — donc clairement au-delà du
débutant — mais pas encore de PRODUIRE SEUL ce volume à ce niveau sans IA. La
marche vers "confirmé" passe par les fondamentaux d'ingénierie manquants (tests,
lint, typage, gestion de la dette).
